# RustyGate v0.4.0

`v0.4.0` makes RustyGate an inference-aware gateway edge for self-hosted OpenAI-compatible replicas. The release focuses on a practical serving problem: repeated long prompt prefixes can waste prefill work when traffic is spread without locality, and overload should be rejected deliberately instead of disappearing into hidden queues.

RustyGate still does not manage model runtime internals. It influences routing, admission, retries, fallback, and observability; the model runtime owns batching, GPU scheduling, prefill/decode execution, KV allocation, and KV eviction.

## Highlights

- Added model pools so one public model ID can represent multiple configured replicas.
- Added admission control for global in-flight requests, model-pool in-flight requests, provider in-flight requests, estimated prompt tokens, and estimated total tokens.
- Added privacy-preserving prefix fingerprinting for stable leading prompt material without logging or persisting raw prompt text.
- Added `prefix_affinity` routing for multi-replica pools, with bounded process-local affinity state, TTL eviction, load-imbalance override, and deterministic tie breakers.
- Added provider-level load and streaming signals, including in-flight counts, queue pressure, TTFT summaries, routing-decision reasons, admission rejection counters, prefix-fingerprint counters, stream outcomes, and stream duration.
- Hardened streaming behavior with `gateway.stream_idle_timeout_ms` for active streams.
- Added route exposure controls for disabling placeholder OpenAI-compatible resource routes in staging or production-style profiles.
- Added shared-prefix benchmark workloads and benchmark configs for comparing priority routing with prefix-affinity routing under mock replicas.

## Migration Notes

Existing provider-only configs continue to load. Adopt the v0.4 features incrementally:

- Add `[[model_pools]]` when several providers are replicas for the same public model. Use `name` and `aliases` for public IDs, `members` for provider names, optional `routing_policy`, and optional `max_in_flight`.
- Set `gateway.routing_policy = "prefix_affinity"` globally, or set `routing_policy = "prefix_affinity"` on a specific model pool.
- Configure `[gateway.prefix_affinity]` with `ttl_seconds`, `max_entries`, `load_imbalance_threshold`, and `fallback_policy`. The fallback policy must be `priority` or `latency`.
- Configure `[gateway.admission]` with optional `max_global_in_flight`, `max_estimated_prompt_tokens`, `max_estimated_total_tokens`, and `retry_after_seconds`.
- Add provider-level `max_in_flight` where a single upstream replica needs its own concurrency cap.
- Keep or tune `gateway.stream_idle_timeout_ms` so stalled upstream streams fail cleanly.
- Set `[gateway.route_exposure].placeholder_compat_routes = false` for internal deployments that should expose only provider-backed inference and model routes.

## Known Limitations

- Admission state, prefix-affinity state, metrics counters, and circuit state are process-local.
- Token budgets use the existing heuristic estimator, not a model-specific tokenizer.
- Rejected requests are not queued in the background; concurrency pressure returns a clean temporary error.
- Prefix affinity is a placement heuristic based on hashed gateway fingerprints. It does not prove runtime prefix-cache or KV-cache reuse.
- The `runtime-cache-signals` Cargo feature remains a mock-backed spike. v0.4 does not ship a real vLLM event, Prometheus, or KV-cache adapter.
- Mock benchmarks measure gateway overhead and routing behavior, not real prefill, decode, batching, GPU execution, or KV-cache speedups.
